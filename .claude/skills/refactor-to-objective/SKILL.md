# refactor-to-objective

Use when the task is to move the code toward an **agreed objective** — the acceptance criterion. The objective (what the toolkit must produce — the 11 stages, FAIR provenance, no-regression rule) is documented in `CLAUDE.md` § **Objective**. Refactor history is in `git log` (commits + diffs). The objective is the thing you converge toward — don't edit it unilaterally; if it needs to change, surface it to the user first. When the objective and the user's in-session input conflict, surface it and prefer the user's input for *requirements*, the objective for *implementation approach*.

This repo (DARDcollect) has tracked debt (god-files and complex functions); the loop below assumes you will keep refactoring opportunistically while preserving behavior. Note: GPU inference (TensorRT/CUDA) is non-deterministic, so the golden gate compares **structure + provenance + schema-validity + volume bounds**, NOT byte-identical inference outputs (see `CLAUDE.md` § No-regression rule + `golden_snapshot.py compare --strict` for the rare deterministic surface).

## Resume protocol (each session)

1. **Recall the objective.** Read `CLAUDE.md` § Objective (the 11 stages, FAIR + Annex IV, no-regression rule) + the relevant sub-doc ([docs/0-GETTING-STARTED.md](docs/0-GETTING-STARTED.md), [docs/1-ARCHITECTURE.md](docs/1-ARCHITECTURE.md), [docs/2-LINEAGE.md](docs/2-LINEAGE.md), [docs/3-ANNOTATIONS.md](docs/3-ANNOTATIONS.md)) as output-format references, NOT the acceptance criterion. For refactor history, `git log`. The local auto-memory (`MEMORY.md`) is loaded automatically each session by the harness — don't hardcode its machine-specific path.
2. **Index first.** Invoke `socraticode-index-first` (SocratiCode index, then codebase_search/flow/impact/graph) before navigating code — don't fan out with grep/glob until the index is ready.
3. **Check tasks.** `TaskList` for in-progress/pending tasks; pick the next one. If empty/stale, rebuild the work list from the objective.
4. **Implement one concrete chunk** — not the whole project. Refactor opportunistically as you go: when the code you're touching (or the code it calls) shows duplication, poor naming, mixed responsibilities, or a missing pattern, improve it in the same chunk. Don't gold-plate unrelated code, but don't leave nearby rot that you just created or stepped through — fix it while the context is loaded. Keep refactors **behavior-preserving** (CSV set, sidecar volume, provenance links, and schema-validity preserved — NOT byte-identical inference, which is GPU-non-deterministic) and **separate from new-feature logic** in the change so the diff reads cleanly.
   - **Do not add runtime fallbacks** (backend cascades, `try/except` that swallows an error and defaults to a degraded path, placeholder models, silent defaults) **without asking the user first** — surface the failing path and let the user choose whether to fall back, fix, or fail loudly. Control over behavior stays with the user, not buried in code. (Note: the repo's automatic GPU→CPU fallback at import is an established, documented behavior — not a new hidden fallback you're adding.)
5. **Dead-code review — every chunk.** While the context for a chunk is loaded, hunt for dead code: unused imports / functions / variables / classes / modules, unreachable branches, commented-out blocks, aspirational stubs never called, leftovers from removed or renamed stages/models. Use SocratiCode `codebase_symbol` (callers/callees) + `codebase_impact` (reverse blast radius) and `ruff` (F401/F811/F841) to find candidates. For each candidate, **investigate *why* it's there before deleting**: grep for references across the repo, check `git log -S` / blame for origin and intent, and weigh public-API (the library API in [docs/5-LIBRARY-API.md](docs/5-LIBRARY-API.md)) or planned-future-use value. If the reason is unclear or it might be deliberate, **ask the user** (surface the symbol, its location, your best guess at origin, and the evidence) — do not delete silently. If it has no justification (genuinely orphaned, leftover, or contradicts current behavior), delete it in the same chunk and prune its references. Log what was removed (and what was kept-after-asking) in the chunk's progress note.
6. **Quality gates — non-negotiable, before marking done:**
   - **Architecture & maintainability:** apply design patterns (strategy/factory/adapter/protocol for swappable backends, etc.), single responsibility, dependency injection where it reduces coupling, naming that reveals intent, no magic numbers. The bar: an architecture reviewer finds a defensible pattern, not a wall of procedural code. This gate is **quantitative, not just prose** — enforce these runnable checks every chunk:
     - **Complexity ratchet (runnable):** `uv run python -m ruff check . --select C901 --config 'lint.mccabe.max-complexity=20' --no-cache`. Measure the violation count at chunk start; the chunk is NOT done if it *increased* the count (non-regression). Reducing the count is progress; the goal is to drive toward `max-complexity=10` over time. (C901 is intentionally NOT in `pyproject.toml` `select` so the main `ruff check` gate stays green while complexity is tracked as ratcheted debt — don't silently move C901 into `select` without fixing the offenders.) Current count as of 2026-07-08: 5 violations (see `CLAUDE.md` § Refactor methodology for the named offenders).
     - **Size discipline (tracked debt):** target functions ≤ ~80 lines and files ≤ ~600 lines. God-files over the target (current as of 2026-07-08, re-measure with `wc -l`): `dardcollect/tracker.py` (~858), `dardcollect/quality.py` (~784), `dardcollect/pipeline_loggers.py` (~748). A chunk must not *grow* a god-file; when you touch one, shrink it (extract coherent units out). New code must stay under the targets.
     - **Layer boundaries (runnable):** `codebase_graph_circular` (SocratiCode) must stay at 0 circular deps (non-regression). Before splitting/renaming, run `codebase_impact` for blast radius. The subsystem→pattern map is maintained in [docs/1-ARCHITECTURE.md](docs/1-ARCHITECTURE.md) and ratified by the user; do not introduce an import that violates it. A hard import-linter gate is a proposed future hardening — not enabled yet, so don't write checks that assume it exists.
   - **Lint + type-check:** run `uv run python -m ruff check .` and `ruff format --check .` (https://github.com/astral-sh/ruff) and `uv run python -m ty check` (https://docs.astral.sh/ty/) — both are configured in this repo's `pyproject.toml`. Fix or consciously suppress findings. The dead-code review above feeds directly into this: F401/F811/F841 findings from ruff are the same dead-code signals. If ruff/ty aren't installed in the env, surface it — don't skip silently.
   - **Documentation:** docstrings on modules/classes/non-trivial functions (purpose, args, returns, raises); *why* comments on non-obvious code; `--help`/usage on new scripts; record design decisions in code AND in the loop memory. Per `keep-docs-navigable`: if a chunk changed a stage, a CLI flag, a config key, a CSV column, a sidecar JSON field, a model, or added an AI system, update the matching sub-doc (and the README AI Systems table + model card) in the same chunk.
   - **Authoring language:** match the surrounding repo content (this repo's authored content is English — keep it English; translate any stray non-English authored text in the same chunk). Don't introduce a second language into the codebase.
7. **Verify before done — objective gate (runnable) + triple-platform, not just tests.**
   - **Objective gate (the loop's acceptance check):** the objective (see `CLAUDE.md` § Objective verification) is met end-to-end when (a) `uv run python scripts/run_pipeline.py --config config.test.yaml` runs the 9 stages on the fast fixture (`tests/fixtures/media/` + `config.test.yaml`, ~1–2 min) with no failed stage, AND (b) `uv run python scripts/golden_snapshot.py --dard-root DARD_test compare tests/fixtures/golden_manifest.json --validate` exits 0. The compare is **non-determinism-tolerant**: it hard-fails on a missing CSV (stage regression), sidecar volume out of bounds, broken provenance, or a schema-invalid sidecar; **drift** (hash diffs / added-or-removed sidecars from GPU non-determinism) is informational and does NOT fail unless `--strict`. The fixture media + `config.test.yaml` + baseline are gitignored/per-machine — build them once with `scripts/make_fixture_media.py` + `scripts/make_test_config.py` + a first `capture` (see CLAUDE.md). If either step fails, the objective is NOT met — fix and re-run; the loop iterates in minutes. The full downloaded dataset is a production run (hours, separate from the dev loop). This gate is the primary verification — a green `pytest tests/` does NOT substitute for it.
   - **Unit suite:** a CPU-only unit suite exists under `tests/` (`test_fair.py`, `test_config.py`). Run `uv run python -m pytest tests/ -q` (no GPU needed). It covers only the pure CPU helpers (FAIR metadata, JSON-Schema validation, config parsing) — the GPU-accelerated stages are verified via the objective gate above, not unit tests. `pytest` is in the `dev` extra (`uv sync --extra dev`). If you add a new pure-CPU helper, add a unit test in the same chunk; if a chunk's surface has no CPU-runnable unit test, rely on the objective gate and surface that honestly — do NOT treat a missing test as a silent pass. The pipeline is GPU-accelerated (ONNX on TensorRT/CUDA or MPS) — **use the GPU when available**.
   - **Golden/snapshot harness:** [scripts/golden_snapshot.py](scripts/golden_snapshot.py) (`capture`/`compare`/`--validate`); baselines live under the gitignored `snapshots/` (machine/dataset-specific). Capture one per machine with `capture` if absent; the user **ratifies** a baseline as the reference. A refactor step is NOT done until the candidate output matches a ratified baseline (or, if behavior is intentionally changing, the user has approved the new baseline) — don't invent a "pass".
   - **No-regression on intentional changes:** when a change is NOT behavior-preserving (an improvement, a fix, a threshold tuning, a model swap), run the relevant stages per `config.yaml` on test media and compare against the **prior** outputs — confirm it is no worse than before (same or more valid detections/crops, provenance chain intact, sidecars still `jsonschema`-valid). A green unit suite does NOT count if a produced artifact or a provenance link regressed. If a metric regressed, the change is not done — revert or fix.
   - **Triple-platform (Linux + Windows + macOS):** the program claims all three — a chunk is not done off a single-platform pass. Run on at least the platforms you have available. If a platform can't be exercised (e.g. no macOS machine), surface that honestly (don't silently skip). Do NOT mark a chunk done claiming cross-platform parity you didn't run.
8. **Update tasks** (`TaskUpdate`) and append a short, dated progress note to the loop memory (most-recent-first, absolute dates).
9. **Stop and request review.** Once a functionality is implemented and the gates pass, PAUSE the loop. Do not auto-continue to the next chunk. **The user is the one who commits — never commit yourself, not even after they approve** (see `CLAUDE.md` § "Working rule — the user commits, never the assistant"). Summarize what changed (files, behavior, lint/type/golden result, gates, platform coverage) and explicitly hand the diff to the user for their review and commit. Do not run `git add`, `git commit`, or `git push` yourself. Wait for the user's decision before resuming — they may want to inspect, commit, adjust scope, or redirect. Only continue to the next chunk when the user says so.

## Scope honesty
Each session does one concrete chunk. Do NOT mark the loop complete until the code demonstrably satisfies the objective (in `CLAUDE.md` § Objective) and the relevant stages run end-to-end on test media producing the expected FAIR artifacts with provenance intact (and, where a baseline exists, a snapshot check confirms unchanged behavior). Be honest about what's **done** vs **blocked by env** (GPU, dataset, missing tooling, missing `tests/`) vs **pending user ratification of a new golden baseline**.

## Blocked-task pivot
When a task is blocked on validation (missing tooling, env, access, or an un-ratified baseline): do NOT burn the loop on unvalidatable hand-written work. Document the gap + fallback strategy in code and memory, keep the task pending/blocked, and pivot to unblocked tasks. Revisit when the blocker lifts (e.g. a new `snapshots/` baseline pending the user running the pipeline on test media).
