# refactor-to-objective

Use to implement or refactor toward the agreed objective — the acceptance criterion. The objective and all quality gates are documented in `CLAUDE.md` (the authoritative single source of truth); this skill describes the **loop protocol** to apply them. 

When the objective and the user's in-session input conflict, surface it and prefer the user's input for *requirements*, the objective for *implementation approach*. Do not edit the objective unilaterally.

This repo has tracked debt (god-files >600 lines, C901 violations); opportunistically refactor while preserving behavior (CSV set, sidecar volume/schema/provenance intact). GPU inference (TensorRT/CUDA) is non-deterministic — the golden gate checks **structure + provenance + validity**, not byte-match (see `CLAUDE.md` § Objective verification).

## Resume protocol (each session)

1. **Recall the objective.** Read `CLAUDE.md` § Objective (the 11 stages, FAIR + Annex IV, no-regression rule) + the relevant sub-doc ([docs/0-GETTING-STARTED.md](docs/0-GETTING-STARTED.md), [docs/1-ARCHITECTURE.md](docs/1-ARCHITECTURE.md), [docs/2-LINEAGE.md](docs/2-LINEAGE.md), [docs/3-ANNOTATIONS.md](docs/3-ANNOTATIONS.md)) as output-format references, NOT the acceptance criterion. For refactor history, `git log`. The local auto-memory (`MEMORY.md`) is loaded automatically each session by the harness — don't hardcode its machine-specific path.
2. **Index first.** Invoke `socraticode-index-first` (SocratiCode index, then codebase_search/flow/impact/graph) before navigating code — don't fan out with grep/glob until the index is ready.
3. **Check tasks.** `TaskList` for in-progress/pending tasks; pick the next one. If empty/stale, rebuild the work list from the objective.
4. **Implement one concrete chunk** — not the whole project. Refactor opportunistically as you go: when the code you're touching (or the code it calls) shows duplication, poor naming, mixed responsibilities, or a missing pattern, improve it in the same chunk. Don't gold-plate unrelated code, but don't leave nearby rot that you just created or stepped through — fix it while the context is loaded. Keep refactors **behavior-preserving** (CSV set, sidecar volume, provenance links, and schema-validity preserved — NOT byte-identical inference, which is GPU-non-deterministic) and **separate from new-feature logic** in the change so the diff reads cleanly.
   - **Do not add runtime fallbacks** (backend cascades, `try/except` that swallows an error and defaults to a degraded path, placeholder models, silent defaults) **without asking the user first** — surface the failing path and let the user choose whether to fall back, fix, or fail loudly. Control over behavior stays with the user, not buried in code.
   - **Fallback exceptions are explicit and finite.** Only the exceptions listed in `CLAUDE.md` "Runtime fallback policy" are pre-approved. If a proposed fallback is not listed there, stop and request user approval before implementing.
5. **Dead-code review — every chunk.** While the context for a chunk is loaded, hunt for dead code: unused imports / functions / variables / classes / modules, unreachable branches, commented-out blocks, aspirational stubs never called, leftovers from removed or renamed stages/models. Use SocratiCode `codebase_symbol` (callers/callees) + `codebase_impact` (reverse blast radius) and `ruff` (F401/F811/F841) to find candidates. For each candidate, **investigate *why* it's there before deleting**: grep for references across the repo, check `git log -S` / blame for origin and intent, and weigh public-API (the library API in [docs/5-LIBRARY-API.md](docs/5-LIBRARY-API.md)) or planned-future-use value. If the reason is unclear or it might be deliberate, **ask the user** (surface the symbol, its location, your best guess at origin, and the evidence) — do not delete silently. If it has no justification (genuinely orphaned, leftover, or contradicts current behavior), delete it in the same chunk and prune its references. Log what was removed (and what was kept-after-asking) in the chunk's progress note.
6. **Quality gates — MANDATORY before marking done.** All gates are documented in `CLAUDE.md` § Objective verification + § Refactor methodology. Run them every chunk; a chunk is NOT done if any gate fails or is not run. The gates:
   - **CPU gates (runnable):** `uv run python -m ruff check .`, `ruff format --check .`, `uv run python -m ty check`, `uv run python -m pytest tests/ -q`, and `uv run python -m ruff check . --select C901 --config 'lint.mccabe.max-complexity=20' --no-cache`. All must pass; C901 count must not increase from chunk start.
   - **Size/Complexity:** files ≤ 600 lines, functions ≤ 80 lines. God-files (tracker.py, quality.py, pipeline_loggers.py) must not grow. C901 must stay ≤ 20 (target 10).
   - **Architecture:** `codebase_graph_circular` (SocratiCode) = 0 circular deps (non-regression). Before splitting/renaming, run `codebase_impact` for blast radius.
   - **Documentation:** docstrings on modules/classes/functions; update README/sub-docs if behavior/config/CSV/sidecar/model/AI system changed (per `keep-docs-navigable`).
   - **Golden gate (objective-gate):** `uv run python scripts/run_pipeline.py --config config.test.yaml` (exits 0, fixture runs) + `uv run python scripts/golden_snapshot.py --dard-root DARD_test compare tests/fixtures/golden_manifest.json --validate` (exits 0, no regression). This is the PRIMARY verification — a green `pytest` does NOT substitute. If either fails, the chunk is NOT done — fix and re-run.
   - **English-only:** match the surrounding repo content (this repo is English-authored — keep all identifiers/comments/text in English).
   - **No silent fallbacks:** do NOT add `try/except` that swallows errors + defaults to a degraded path without asking the user first.
   - **Exception discipline:** if you touch an existing approved fallback, keep it logged/observable; if fallback behavior changes, update `CLAUDE.md` rules + relevant docs in the same chunk.
7. **Verify before done — objective gate (runnable) is MANDATORY, not optional.** See `CLAUDE.md` § Objective verification for the complete gate definition and setup steps. In short:
   - **Step 1:** `uv run python scripts/run_pipeline.py --config config.test.yaml` — runs all 11 stages on the fast fixture (~1–2 min); must exit 0.
   - **Step 2:** `uv run python scripts/golden_snapshot.py --dard-root DARD_test compare tests/fixtures/golden_manifest.json --validate` — compares output structure/provenance/schema-validity (non-determinism-tolerant, GPU drift is informational); must exit 0.
   - If either step fails, the chunk is NOT done — fix and re-run; iterate in minutes.
   - **Triple-platform:** if claiming cross-platform parity, test on Linux + Windows + macOS. If a platform can't be exercised, surface that honestly (don't silently skip).
8. **Update tasks** (`TaskUpdate`) and append a short, dated progress note to the loop memory (most-recent-first, absolute dates).
9. **Stop and request review.** Once a functionality is implemented and the gates pass, PAUSE the loop. Do not auto-continue to the next chunk. **The user is the one who commits — never commit yourself, not even after they approve** (see `CLAUDE.md` § "Working rule — the user commits, never the assistant"). Summarize what changed (files, behavior, lint/type/golden result, gates, platform coverage) and explicitly hand the diff to the user for their review and commit. Do not run `git add`, `git commit`, or `git push` yourself. Wait for the user's decision before resuming — they may want to inspect, commit, adjust scope, or redirect. Only continue to the next chunk when the user says so.

## Scope honesty (§ Chunk DONE vs NOT DONE in CLAUDE.md)

Each session does one concrete chunk. The checklist of "chunk DONE" vs "chunk NOT DONE" is explicit in `CLAUDE.md` § Scope honesty — use it as a gate. Critical rule: **if the objective gate is pending or failing, the chunk is NOT done**, no matter how green the CPU gates are. Surface the status honestly to the user — "CPU gates ✅, objective gate 🔄 IN PROGRESS" or "objective gate ❌ FAILED (reason)" — do not invent a "done" state.

## Blocked-task pivot
When a task is blocked on validation (missing tooling, env, access, or an un-ratified baseline): do NOT burn the loop on unvalidatable hand-written work. Document the gap + fallback strategy in code and memory, keep the task pending/blocked, and pivot to unblocked tasks. Revisit when the blocker lifts (e.g. a new `snapshots/` baseline pending the user running the pipeline on test media).
